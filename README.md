# Harness Study

## Antigravity IDE에서의 Harness Engineering

### 슬래시 명령어

Antigravity IDE의 대화 인터페이스에서 슬래시(`/`)를 입력하여 특정한 목적의 자동화 워크플로나 에이전트의 맞춤형 동작을 유도할 수 있는 명령어 목록입니다.

- **`/goal`**: 긴 시간 동안(예: 밤샘 빌드, 대규모 리팩토링 등) 수행해야 하는 작업을 에이전트에게 지시할 때 사용합니다. 에이전트가 목표가 완전히 완수될 때까지 중간에 멈추지 않고 더 철저하게 검증하도록 강제합니다.
- **`/schedule`**: 일회성 타이머를 설정하거나 주기적인 작업(Cron 식 활용)을 설정하여 특정 시각에 에이전트가 자동으로 알림을 보내고 관련 작업을 시작하게 할 때 사용합니다.
- **`/grill-me`**: 개발을 시작하기 전, 설계 결정이나 모호한 요구사항을 맞추기 위해 에이전트와 인터랙티브한 대화식 인터뷰를 진행하고 최선의 계획을 조율하고자 할 때 사용합니다.
- **`/learn`**: 에이전트의 잘못된 동작을 수정했거나 프로젝트의 특이한 빌드/개발 환경 설정을 조율한 뒤, 이 노하우를 에이전트가 앞으로의 대화에서도 지속적으로 기억하고 활용하도록 영속화시킬 때 사용합니다.

### MCP Server (Model Context Protocol Server)

MCP(Model Context Protocol) 서버는 Antigravity IDE 내부의 AI 에이전트가 외부 서비스나 로컬 파일/도구에 안전하게 접속하여 양방향 데이터를 주고받을 수 있게 해주는 프로토콜 허브입니다.

#### 1. 연결 및 설정 방법

- **전역 설정 파일 편집**: `~/.gemini/config/mcp_config.json` 경로에서 설정합니다. 이 설정은 IDE, CLI 등 모든 Antigravity 환경에서 공유됩니다.
- **IDE 설정 UI 활용**: IDE 우측의 **Agent panel** 상단의 `...` (옵션) 버튼 클릭 → `MCP Servers` → `Manage MCP Servers`를 통해 설정 및 조작할 수 있습니다.
- **설정 변경 반영**: 설정을 마친 후 `Manage MCP Servers` 창에서 **Refresh** 버튼을 눌러 변경된 내용을 에이전트 환경에 적용합니다.

#### 2. 설정 예시 (`mcp_config.json`)

```json
{
  "mcpServers": {
    "custom-mcp-server": {
      "command": "node",
      "args": ["/absolute/path/to/server/index.js"],
      "env": {
        "API_KEY": "your-secret-api-key"
      }
    }
  }
}
```

#### 3. 로컬 vs 원격 서버 설정 차이점

- **로컬 서버**: 일반적으로 로컬 머신에서 작동하며 표준 입출력(`stdio`) 방식으로 통신합니다.
- **원격 서버**: HTTP 엔드포인트를 통해 통신할 경우, 설정에서 `url` 필드 대신 `serverUrl` 필드를 명시하여 엔드포인트를 지정합니다.

### Artifact

아티팩트는 에이전트가 복잡한 분석 결과, 작업 목록, 또는 개발 설계서 및 검증 보고서 등을 구조화하여 대화창 외부의 별도 문서로 제공하는 정적인 산출물입니다. 주로 `<appDataDir>/brain/<conversation-id>/` 경로 밑에 Markdown(`.md`) 형식으로 저장됩니다.

#### 핵심 아티팩트 3가지

1. **Task List (`task.md`)**
   - **설명**: 작업의 구체적인 구현 단계와 체크리스트를 관리하는 할 일 목록입니다.
   - **상태 표기**: 진행 상태에 따라 `[ ]` (미완료), `[/]` (진행 중), `[x]` (완료) 상태를 실시간으로 관리하며 진행율을 트래킹합니다.
2. **Implementation Plan (`implementation_plan.md`)**
   - **설명**: 기능 변경이나 시스템 확장에 앞서 에이전트가 사용자에게 명시적으로 승인을 받기 위해 제출하는 상세 설계 문서입니다.
   - **포함 정보**: 목표 설명, 사용자 검토가 필수적인 부분(디자인 변경, 중요 의사결정 등), 미결 질문(Open Questions), 파일별 세부 수정 사항(`[MODIFY]`, `[NEW]`, `[DELETE]`), 자동/수동 검증 계획.
3. **Walkthrough (`walkthrough.md`)**
   - **설명**: 기능 구현 및 리팩토링이 끝난 후, 어떤 것들이 변경되었고 어떻게 테스트 및 검증이 되었는지를 한눈에 보여주는 완성 보고서입니다.
   - **포함 정보**: 최종 구현 설명, 테스트 및 검증 결과, 필요에 따라 실행 화면이나 UI 변화를 증명하는 스크린샷과 비디오 녹화본.

#### 아티팩트의 특징

- 경고/정보 제공을 위한 GitHub Alert 스타일(`> [!NOTE]`, `> [!IMPORTANT]` 등) 및 Mermaid 다이어그램을 활용하여 흐름을 시각화합니다.
- 여러 스크린샷이나 코드를 순차적으로 보여주기 위한 Carousel 구조(`carousel ... `)를 활용할 수 있습니다.

### 지정된 폴더 구조

Antigravity IDE는 개발 생산성 극대화 및 하네스 엔지니어링을 위해 프로젝트 내에 정형화된 폴더 및 파일 구조 규칙을 제공합니다.

#### 1. 커스터마이징 루트 (Customization Roots)

- **전역 커스텀 루트**: `~/.gemini/config/`
- **프로젝트(작업 공간) 커스텀 루트**: 프로젝트 루트의 `.agents/` 디렉터리

#### 2. 에이전트 작동 규칙 정의 - `AGENTS.md`

에이전트가 행동할 때 지켜야 할 코딩 스타일 가이드, 보완 규칙, 제약 사항을 정의하는 파일입니다.

- 전역으로 적용할 규칙은 `~/.gemini/config/AGENTS.md`에 기록합니다.
- 특정 프로젝트에만 적용할 규칙은 `.agents/AGENTS.md`에 기술하여 프로젝트별로 다르게 가이드라인을 제공할 수 있습니다.

#### 3. 스킬 정의 및 폴더 구조 - `skills/`

특정한 실행 도구나 시나리오별 명령어를 담은 '스킬'을 패키징하는 곳입니다. `.agents/skills/<skill_name>/` 경로에 생성합니다.

- **`SKILL.md` (필수)**: YAML Frontmatter 형식을 포함한 마크다운 파일로, 에이전트가 이 스킬을 탐색할 수 있는 `name`과 `description` 정보가 기재되어 있어야 합니다.

  ```markdown
  ---
  name: "custom_build_tool"
  description: "Runs and verifies the custom build scripts for this project."
  ---

  (스킬의 사용법과 구체적인 행동 가이드 기재)
  ```

- **하위 폴더 구조 예시**:
  - `scripts/`: 스킬 실행에 동원되는 자동화 헬퍼 스크립트들
  - `examples/`: 참고할 수 있는 스킬 구현 예제 코드
  - `resources/`: 스킬 동작에 필요한 설정 파일이나 정적 데이터
  - `references/`: 스킬의 상세 원리나 개념을 이해하는 데 필요한 기술 문서
- **`skills.json` (외부 경로 스킬 등록 시)**: `.agents/` 디렉터리가 아닌 외부 공유 폴더의 스킬들을 연동할 경우, `skills.json`에 `entries`, `inherits`, `exclude` 필드를 지정해 스킬들을 수동으로 불러올 수 있습니다.

#### 4. 기능 정의 및 요구사항 폴더 - `.agents/docs/specs/` 또는 `docs/`

에이전트가 신규 기능을 구현하거나 기존 기능을 변경할 때 가장 먼저 읽고 참고하는 기능 요구사항 명세서(Feature Specification)들이 보관되는 디렉터리입니다.

- **역할**: 구현할 기능의 상세 배경, 비즈니스 로직, 기능별 수용 기준(Acceptance Criteria), 입력/출력 사양 등을 마크다운 파일(예: `spec_dino_movement.md`, `feature_auth.md`)로 작성하여 에이전트의 작업 범위를 정의합니다.
- **효과**: 에이전트는 구현 전 이 폴더 내의 사양서를 통해 기획 의도를 온전히 이해하므로, 요구사항 누락 없이 정확한 목표를 바탕으로 코드를 생성할 수 있습니다.

#### 5. 디자인 설계 및 아키텍처 문서 - `.agents/docs/design/` 또는 `design_notes.md`

UI/UX 디자인 가이드라인, 컴포넌트 구조도, 데이터 흐름도 및 시스템 아키텍처 정보를 담은 문서입니다.

- **역할**: UI 목업 디자인 규칙(레이아웃, 색상 테마, 타이포그래피, 정렬 방식 등)이나 컴포넌트/클래스 간의 관계를 상세히 기술합니다. 특히 Mermaid 다이어그램을 활용해 상태 전이(State Machine)나 아키텍처 구조를 시각화해두면 에이전트의 이해도가 향상됩니다.
- **효과**: 에이전트가 기존 시스템의 설계 방향이나 UI 디자인 가이드를 일관되게 적용하게 함으로써, 코드베이스의 아키텍처적 일관성을 유지할 수 있습니다.

#### 6. 개발 로그 및 실행 기록 보관 - `.agents/logs/` 또는 `logs/`

에이전트가 개발 및 테스트를 진행하면서 발생한 테스트 결과, 디버깅 로그, 빌드 에러 출력물 등을 기록하고 관리하는 공간입니다.

- **역할**: 빌드 실패 로그, 테스트 커버리지 리포트, 런타임 에러 추적 스택 등을 텍스트 파일로 저장하여 트래킹합니다.
- **추가 팁 (Scratch 디렉터리)**: 에이전트가 세션을 진행하며 임시로 생성하는 디버깅 스크립트나 로컬 검증 데이터는 대화 세션 전용 임시 디렉터리인 `<appDataDir>/brain/<conversation-id>/scratch/` 경로에 저장되어 에이전트의 자가 디버깅(Self-debugging)의 원천 데이터로 사용됩니다.

#### 7. 그 외 에이전트가 참고하는 정보

- **Knowledge Items (KI) 시스템**: 에이전트가 대화 세션을 시작할 때 자동으로 로드하는 로컬 지식 스냅샷으로, `<appDataDir>/knowledge/`에 위치합니다. 과거 세션에서 해결된 고난도 버그, 특정 라이브러리의 프레임워크 설정 팁, 권장/비권장 API 패턴 등이 아티팩트(`metadata.json`, `artifacts/`) 형태로 정형화되어 있어 에이전트가 동일한 실수나 중복 작업을 방지하는 데 필수적인 리소스로 활용됩니다.
- **스크래치 스크립트 (Scratch Scripts)**: 일회성으로 API 통신을 확인하거나 DB 스키마 등을 간단히 검사하기 위해 작성한 스크립트는 프로젝트 메인 소스코드를 어지럽히지 않도록 에이전트 전용 scratch 디렉터리에 저장해 두고 검증에 활용합니다.
